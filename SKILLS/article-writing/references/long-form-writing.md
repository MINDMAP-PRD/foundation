# Long-Form Writing

Use this reference when a task needs polished long-form content rather than a quick summary.

## 1. Voice Capture Workflow

If the user wants a specific voice, collect one or more of:

- published articles
- newsletters
- posts
- docs or memos
- a short style guide

Extract:

- sentence length and rhythm
- whether the voice is formal, conversational, sharp, or restrained
- favored rhetorical devices such as fragments, questions, parentheses, lists, or pull quotes
- tolerance for humor, opinion, and contrarian framing
- formatting habits such as headers, bullets, code blocks, and section breaks

If no voice references are provided, default to a direct operator-style voice:

- concrete
- practical
- low on hype
- credible

## 2. Banned Patterns

Delete and rewrite any of these:

- generic openings such as "In today's rapidly evolving landscape"
- filler transitions such as "Moreover" or "Furthermore"
- hype language like "game-changer", "cutting-edge", or "revolutionary"
- vague claims without evidence
- credibility claims not backed by supplied context

## 3. Writing Process

1. Clarify the audience and purpose.
2. Build a skeletal outline with one purpose per section.
3. Start each section with evidence, example, or scene.
4. Expand only where the next sentence earns its place.
5. Remove anything that sounds templated, padded, or self-congratulatory.

## 4. Structure Guidance

### Technical Guides

- open with what the reader gets
- use code or terminal examples in major sections when appropriate
- end with concrete takeaways, not a vague soft summary

### Essays Or Opinion Pieces

- start with tension, contradiction, or a sharp observation
- keep one argument thread per section
- use examples that earn the opinion

### Newsletters

- make the first screen strong
- mix insight with updates
- use clear section labels and skim-friendly structure

## 5. Grounding Rule

If the writing is based on repo work, package docs, audits, or technical sources:

- verify factual claims against the provided materials
- do not invent missing evidence
- explicitly leave unsupported claims out instead of guessing

## 6. Quality Gate

Before delivering:

- verify factual claims against provided sources
- remove filler and corporate language
- confirm the voice matches the supplied examples when examples exist
- ensure every section adds new information
- check formatting for the intended platform
